# Platform gotchas

## Check Rich Text Area content with `LEN() > 0`

Use `LEN(fieldName) > 0` (not `ISBLANK` or `ISNULL`) to check whether a Rich Text Area field has content in validation rules and formula fields.

**Why:** `ISBLANK()` always returns `false` for Rich Text Area fields (Salesforce explicitly does not support it). `ISNULL()` also fails to detect cleared Rich Text Area fields reliably. `LEN() > 0` correctly handles all states — null, empty string, and cleared-via-API values.

## `cacheable=true` is served from the platform cache even imperatively

A `cacheable=true` Apex method called via LWC `@wire` is served from the platform's client-side cache across component instances, even after the component fully unmounts and remounts (e.g. navigating away and back via an `if:true` toggle).

**Correction (live-org evidence):** `cacheable=true` methods can be served from the storable-action cache even when called **imperatively** — it is NOT true that imperative calls always bypass it. The fix that actually works is to drop `cacheable=true` entirely from any method that is never consumed via `@wire`. The "imperative bypasses the cache" theory cost two failed fix attempts before live debugging (console logging through the actual create/delete flow) proved the call kept returning a stale row count regardless of how it was triggered.

**How to apply:** For any LWC list/queue that must reflect writes made elsewhere and always be fresh on view: if the method is never consumed via `@wire` anywhere, remove `cacheable=true` from it entirely — that is the only verified-working fix. Confirm with live testing. Pair with Lightning Message Service if writer and list live in different component trees, and/or `CurrentPageReference` if returning from another page should also trigger a reload. `refreshApex` is the other invalidation route but needs a wire result reference populated in the _same_ instance, which is racy from `connectedCallback`.

## Lightning Message Service needs `APPLICATION_SCOPE` across page regions

`subscribe()` calls need `{ scope: APPLICATION_SCOPE }` (imported from `lightning/messageService`) as the 4th argument whenever publisher and subscriber live in different regions of a Lightning page — e.g. a Utility Bar component publishing to a component on the App Page body. The default scope only delivers within the same DOM/component region.

**Why:** Caught in code review — a fix used LMS to notify an App Page component of a utility bar action but called `subscribe()` without the scope option. Jest's `lightning/messageService` stub (`subscribe`/`publish` are plain `jest.fn()`s) has no real scoping behaviour, so unit tests pass even when the cross-region message would never arrive at runtime. Only code review or manual org testing catches it.

**How to apply:** Any time LMS bridges a Utility Bar item and an App Page (or any two components not in the same subtree/region), pass `{ scope: APPLICATION_SCOPE }` to `subscribe()`. Default scope is fine only when both ends share the same flexipage region/component tree.

# Metadata, Prettier, and deploy

## Never include `xml` in Prettier globs

Salesforce metadata files (permission sets, flows, objects, page layouts) are generated by the CLI or managed in the Salesforce UI. Prettier reformats them in ways that cause spurious diffs and break org ↔ source sync. Remove `xml` from both `scripts.prettier` and `lint-staged` in `package.json`.

Prettier should cover only human-authored code:

```
cls, trigger        — Apex
html, js, css       — LWC
cmp, component, page — Visualforce/Aura
json, md            — config and docs
yaml, yml           — CI/tooling config
```

**Why:** Prettier reformatted description elements across multiple lines and reordered tab settings — diffs unrelated to real changes that made the org retrieve show "Changed" on every commit.

## VS Code XML extension — suppressing metadata false-positives

In `.vscode/settings.json`:

```json
"xml.validation.noGrammar": "ignore",
"xml.downloadExternalResources.enabled": false,
"xml.validation.filters": [
  { "pattern": "**/*-meta.xml", "enabled": false }
]
```

All three are needed:

- `noGrammar: ignore` — suppresses errors when a file has no schema. Not enough alone because Salesforce metadata files declare `xmlns=...`, which the extension treats as "grammar found."
- `downloadExternalResources.enabled: false` — stops the extension fetching Salesforce XSDs from the namespace URL.
- `xml.validation.filters` — the definitive fix: disables validation entirely for all `-meta.xml` files. Takes effect immediately, no cache clear or reload. "XML: Clear Schema Cache" is not a reliable command — it may not exist depending on extension version.

**Anti-patterns that made it worse:**

- **`systemId: ""`** — does NOT mean "no schema." It means "use an empty external DTD," which triggers DTD validation mode and flags every element as undeclared (130 errors instead of 1). Never use it to suppress validation.
- **Local XSD with `targetNamespace`** — a local `.xsd` declaring `targetNamespace="http://soap.sforce.com/2006/04/metadata"` makes the extension apply that namespace's cached schema to ALL files sharing it, including unrelated permission sets. This surfaced a broken cached Salesforce XSD (`salesforce_metadata_api_common.xsd`, with a `MetadataForSettings` type reference error) as a false positive on permission set files. Fix: remove the fileAssociation and rely on `noGrammar` + `downloadExternalResources`.

## Permission set deployment — required fields

Required fields (`<required>true</required>`) cannot be included in a permission set's `<fieldPermissions>`. Salesforce rejects the deploy with "You cannot deploy to a required field: `Object.Field__c`." Simply omit required fields — they are automatically accessible to anyone with object access. (Required fields must always be writable; the permission system doesn't need to grant what's already guaranteed.) For the per-object permission-set mapping, see [[project]].

## Lock app navigation personalisation

When creating a `CustomApplication` (Lightning App), always set:

```xml
<isNavPersonalizationDisabled>true</isNavPersonalizationDisabled>
```

**Why:** Leaving this `false` (the Salesforce default) lets users add, remove, and reorder nav tabs for themselves; their personal customisation then hides tabs that should be there, which is confusing to troubleshoot and defeats a controlled app experience.

**How to apply:** Set it in every `.app-meta.xml` at creation time — it is not the default, so it must be explicit. If an existing app has `false`, update the source and deploy.

## Lightning Record Pages for new custom objects

Salesforce defaults to the classic page layout for any new custom object, even if a Lightning Record Page exists in source. The org-default activation cannot be set via the Metadata API or Tooling API — it must be done manually.

**Rule:** Every time a new custom object is created, also create a Lightning Record Page (`*.flexipage-meta.xml`, type `RecordPage`) for it and deploy it.

**Human action required after deploy:** Open the org → a record of the new object → gear icon → Edit Page → Activation → Assign as Org Default → Save. One-time step per org; cannot be automated. The assignment is stored internally and is not written back to the FlexiPage XML.

# MCP tooling

## Which MCP server owns what

`sf-project-cli` (the Salesforce dev-tooling MCP server) only exposes SOQL query, metadata retrieve/deploy, DevOps Center, and code-analyzer tools — **no generic SObject create/update tool**.

Record DML for this project's six custom objects (`Work_Item__c`, `Sprint__c`, `Documentation__c`, `Change_Log__c`, `Comment__c`, `Folder__c`) lives on a **separate** MCP server, `salesforce-project-doc`:

- `ProjectMCPCreateRecord` — `fieldsJson` + `objectApiName`
- `ProjectMCPUpdateRecord` — `fieldsJson` + `recordId`
- `ProjectMCPGetObjectSchema` — field metadata for the six objects
- `ProjectMCPGetRelatedRecords` — needs `childObjectApiName`, `parentId`, `parentLookupField` (all three, no defaults)

**Why:** Got blocked mid-pipeline (Code Review Agent trying to log a `Comment__c` and update `Status__c`) because `sf-project-cli` genuinely has no write tool. If `salesforce-project-doc` tools aren't showing in ToolSearch results, the connection may need reauth — ask the user rather than falling back to CLI/Apex.

## Always prefer MCP over alternatives

MCP is the first and only choice for any Salesforce data read or write. Do not use anonymous Apex scripts (`sf apex run` — creates temp files, dirties git status), PowerShell `Invoke-RestMethod` / `sf api request rest`, or `sf data record create`. Reserve non-MCP routes only for things MCP genuinely cannot do: governor-limit-sensitive bulk DML across tens of thousands of records, or transactional all-or-nothing semantics across many objects in a single Apex context.

## Confirmation rules

Read tools always fire without asking — proceed directly: `soqlQuery`, `getObjectSchema`, `getRelatedRecords`, `getUserInfo`, `listRecentSobjectRecords`, `find`. Write tools (`createSobjectRecord`, `updateSobjectRecord`, `updateRelatedRecord`) warrant narration before firing per [[how_i_work]]. (The user explicitly stated reads are always allowed; asking adds friction.)

## Record creation patterns

- **Record Type — always pre-query for Id.** Relationship notation (`"RecordType": {"DeveloperName": "Update"}`) is rejected with "DeveloperName is not an External ID or indexed field for RecordType." Pre-query `SELECT Id FROM RecordType WHERE SobjectType = '...' AND DeveloperName = '...'` and pass the explicit `RecordTypeId`.
- **`Comment__c.Body__c` — always plain text.** The `workItemComments` LWC renders it as plain text (no rich-text component). HTML tags are stored literally and appear as raw tag text in the UI.
- **Large Rich Text Area fields on other objects — two-step create then update.** For Rich Text Area fields on objects other than `Comment__c`, the Salesforce API can silently drop content on `createSobjectRecord` when payloads are large. Fix: create with metadata fields only, then `updateSobjectRecord` the new record Id to set the Rich Text Area fields. Long Text Area fields are not affected.

# Rich Text Area — the canonical limit and how to write it

## The cap is fixed at 32,768 chars

`Documentation__c.Body__c` and the rich-text fields on `Change_Log__c` (`Staged_Technical_Body__c`, `Staged_User_Body__c`, plus the legacy Before/After snapshot fields) are `Rich Text Area(32768)`. **32,768 chars is Salesforce's hard platform ceiling for this field type — it cannot be raised.** This is the single source for this number; other files reference it.

**Size discipline:** For a small targeted edit (one paragraph/bullet), don't pre-check — just attempt the update (the error is more authoritative than any local check, per [[how_i_work]]). But before any **full-document rewrite or backfill**, check the candidate HTML length first (e.g. `node -e "console.log(fs.readFileSync(path).length)"`) against 32,768. A repo doc may have outgrown the cap — `docs/technical/project-management-guide.md` grew to ~56KB markdown (~60KB+ HTML), 2x over. If the repo file is large, the SF baseline is almost certainly a hand-condensed prose summary, not a 1:1 conversion — match the SF baseline's density, don't mechanically convert the whole repo file. See [[docs]].

## Escaping large payloads via `fieldsJson`

Never hand-type a large (1K+ char) HTML/Rich Text Area string directly into a `fieldsJson` tool-call parameter. Generate it programmatically and verify the escaping round-trips before sending.

**Why:** `fieldsJson` is double-JSON-encoded — the tool-call arguments are JSON, and `fieldsJson`'s value is itself JSON text. A literal `"` needs `\\\"` (4 chars) in the source; a literal newline needs `\\n` (3 chars). A bare `\n` or `\"` gets consumed by the outer parse and produces a raw control character (illegal in JSON) or a premature string terminator. Manually tracking this across a 19K-char payload caused 4+ failed attempts in one session with cryptic parser errors (`Illegal unquoted character`, `Unexpected character... was expecting comma`).

**How to apply:**

1. Write the target HTML to a local scratch file with the `Write` tool (preserves literal characters exactly).
2. Replace raw newlines with the HTML entity `&#10;` before JSON-encoding — sidesteps the newline-escaping problem entirely (no functional difference when rendered).
3. Build the payload via a small Node script: `JSON.stringify({ Field__c: html })` (= STRING_B), sanity-check with `JSON.parse(STRING_B)`, then `JSON.stringify(STRING_B).slice(1, -1)` for the exact text to paste as `fieldsJson`. (Equivalently: `node -e "process.stdout.write(JSON.stringify('<content>'))"`.)
4. Read the generated file and paste its _exact_ content — do not retype, "clean up," or substitute any character (especially em-dashes, quotes). Retyping from memory caused repeated failures even after the correct pattern was known.
5. If unsure the escaping is right, validate with a tiny isolated test first (update a scratch field with a 20-char string containing a quote) before risking the full payload.
6. Delete scratch files after use.

See [[docs]] for the related bare-field-name corruption gotcha in the _source_ Markdown before it reaches this escaping step.
